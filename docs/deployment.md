# Pyxle Deployment Guide

Pyxle ships two complementary workflows:

1. **`pyxle dev`** — Launches the bundled dev server with Vite, Starlette, the file watcher, and the overlay.
2. **`pyxle build`** — Compiles your project for production and emits distributable artifacts under `dist/` (or a custom folder).

This guide walks through the prerequisites, the recommended development startup flow, and the current production build pipeline.

## 1. Prerequisites

| Component | Recommended Version | Notes |
|-----------|--------------------|-------|
| Python    | 3.12+              | Matches the interpreter used to run the CLI/build pipeline. |
| Node.js   | 20.x LTS           | Needed for the Vite dev server and production bundling. |
| pip / npm | Latest stable      | Used to install the Python + JS dependencies generated by `pyxle init`. |

After creating a project (`pyxle init my-app`), install dependencies:

```bash
pyxle install
```

The command runs `python -m pip install -r requirements.txt` and `npm install` inside
your project directory; use `--no-python` / `--no-node` to skip either step or run the
individual commands manually if you prefer.

## 2. Running the development server (`pyxle dev`)

`pyxle dev` transpiles `.pyx` pages, copies API modules, launches Vite, and starts the Starlette app with hot reload. Typical flags:

```bash
pyxle dev \
  --host 0.0.0.0 \
  --port 8000 \
  --vite-host 127.0.0.1 \
  --vite-port 5173 \
  --log-format console \
  --print-config
```

Key behaviors:

- The command reads `pyxle.config.json` (override with `--config`). CLI flags win over file settings.
- Readiness endpoints: `/healthz` always reports uptime, `/readyz` flips to `200` after Vite + the watcher finish booting.
- Logs can be emitted in **console** or **json** format (`--log-format json`) for observability pipelines.
- File changes under `pages/` or `public/` trigger incremental rebuilds automatically.

For staging-like environments, run `pyxle dev --no-debug` to disable overlay noise and to emit production-style logs while retaining hot reload.

### Injecting custom middleware

Set the `middleware` array in `pyxle.config.json` to register Starlette middleware classes that should wrap every request:

```jsonc
{
  "middleware": [
    "middlewares.custom:PyxleCustomHeaderMiddleware",
    "middlewares.auth:RequireAuthMiddleware"
  ]
}
```

- Each entry follows `"module:attribute"`.
- Modules are resolved relative to the project root (Pyxle adds the root to `sys.path` before loading middleware), so you can keep implementations under `middlewares/` without creating a package.
- Attributes can be `BaseHTTPMiddleware` subclasses, a `starlette.middleware.Middleware` instance, or a callable returning either of those — mirroring the behavior covered in `tests/devserver/sample_middlewares.py`.

Restart the dev server after updating the config and hit any route (e.g. `/api/pulse`) to ensure headers or other side effects are present. The same configuration is honored by the production build pipeline.

## 3. Building production artifacts (`pyxle build`)

When you are ready to deploy, run the production pipeline:

```bash
pyxle build \
  --config pyxle.config.json \
  --out-dir dist
```

What the command does:

1. Regenerates `.pyxle-build/` (compiles pages + copies APIs).
2. Writes the client bootstrap files required by Vite.
3. Copies the server modules, metadata, and public assets into `dist/`.
4. Executes a Vite production build, emitting hashed bundles into `dist/client`.
5. Generates the Pyxle page manifest (used for future SSR + asset lookup).

When the build finishes, the CLI lists the most important artifact locations so CI logs double as a deployment checklist:

- `Client manifest` — Vite's `manifest.json` inside `dist/client/`.
- `Page manifest` — `dist/page-manifest.json`, which maps every compiled route to its metadata file, server module, and bundled client assets.
- `Server modules` — `dist/server/`.
- `Metadata` — `dist/metadata/`.
- `Public assets` — `dist/public/` (if your project ships a `public/` directory).
- `Artifacts` — the resolved `dist/` (or the directory provided via `--out-dir`).

`dist/` layout:

```
dist/
├── client/      # Hashed JS/CSS bundles from Vite
├── public/      # Copied static assets (favicon, images, etc.)
├── server/      # Python modules generated from your `.pyx` files
├── metadata/    # Route + loader metadata consumed by the runtime
├── manifest.json        # Vite output consumed by the client runtime
└── page-manifest.json   # Pyxle manifest linking routes to metadata + bundles
```

### Serving hashed assets at runtime

- When you disable debug mode (`pyxle dev --no-debug` or `"debug": false` in `pyxle.config.json`), the SSR renderer skips the Vite dev server and reads `dist/page-manifest.json` to discover the hashed client bundle(s) for each route.
- The Starlette app automatically loads that manifest on startup, so make sure `pyxle build` has been executed and the entire `dist/` folder (including `page-manifest.json`) is deployed alongside your server modules.
- Missing or stale manifest data falls back to a plain HTML warning encouraging you to rebuild, so you can quickly detect when the production assets are out of sync.

Host `dist/client` via your CDN or web server, and bundle `dist/server` + `dist/metadata` with your ASGI application container. (A dedicated `pyxle serve` target is on the roadmap; for now the server package mirrors the development runtime.)

### Continuous integration tips

- Run `pyxle build --out-dir dist` inside CI to catch Vite/SSR regressions early.
- Archive the entire `dist/` folder as an artifact so deploy environments do not need Node installed.
- Fail the pipeline if `pyxle build` exits non-zero — Vite and the compiler bubble actionable errors.

## 4. Starting the server in production today

Until the dedicated production runner lands, the recommended approach is to run `pyxle dev` behind a process manager with debug features disabled:

```bash
PYXLE_ENV=production pyxle dev \
  --host 0.0.0.0 \
  --port 8080 \
  --vite-port 4173 \
  --no-debug \
  --log-format json
```

Guidelines:

- Run under `systemd`, `supervisord`, or a container platform so the process auto-restarts.
- Terminate TLS at a reverse proxy (NGINX, Caddy, Fly.io, etc.) and forward traffic to `pyxle dev`.
- Use the `/readyz` endpoint as a readiness probe so load balancers wait for Vite to come online.
- Configure observability by pointing structured logs to your log shipper when using `--log-format json`.

## 5. What’s next

The roadmap tracks an upcoming `pyxle serve` command that will boot the Starlette app directly from `dist/` without Vite. The documentation above will evolve accordingly; for now the combination of `pyxle build` + `pyxle dev --no-debug` covers both preview and production deployments.
